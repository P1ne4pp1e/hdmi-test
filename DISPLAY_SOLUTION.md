# HDMI 工控显示方案

## 1. 目标

本项目验证一套适用于小型 Linux 工控设备的 HDMI 显示方案：设备开机后不启动传统桌面环境，直接显示一个全屏、现代化、可交互的 HMI（Human-Machine Interface）。

界面由 C++ 主导，包含现代化 UI、按钮与状态信息，以及由程序持续绘制的动态画面。未来更换触摸屏后，应能直接接入触摸操作。

## 2. 核心结论

DMA 本身不是“渲染最快”的替代品；它是让图像缓冲区在硬件模块之间传递时**避免 CPU 逐像素复制**的机制。最快的完整路径通常是：

```text
GPU/硬件生产图像 ── DMA-BUF ── GPU 合成或 KMS Plane ── HDMI
```

DMA-BUF 的价值是减少拷贝和延迟。它不能替代 UI 布局、GPU 绘制、DRM/KMS 模式设置或 HDMI 输出控制。

对本项目，推荐的正式基础是：

```text
systemd → Qt 6 Quick（C++ 后端）→ EGLFS/DRM-KMS → HDMI
```

需要逐帧显示的画面作为 Qt Quick 页面中的动态渲染区域；仅在该画面来自支持 DMA-BUF 的设备或硬件模块时，再接入零拷贝 DMA-BUF 路径。

### 当前 Jetson 平台验证结果

目标设备为 Jetson Orin Nano Developer Kit，JetPack 6.2.2（L4T R36.5）。当前 HDMI 屏幕在 NVIDIA 专有 X 驱动中已正常工作：X RandR 显示 `DP-1 connected primary 800x480`，`SignalFormat: TMDS`，并读到了 LONTIUM EDID。这里的 `DP-1` 是 NVIDIA 驱动对输出端的命名，TMDS 表明其当前使用 HDMI 信号链路。

但是通用 `libdrm` 对 `/dev/dri/card1` 的 connector 枚举同时得到 `DP-1: disconnected, 0 mode(s)`。因此，通用 DRM dumb-buffer 程序无法直接替代当前 NVIDIA 显示栈；这不是线缆、显示器休眠或权限问题。后续必须验证与 JetPack NVIDIA 驱动兼容的无桌面后端（例如 NVIDIA 支持的 Wayland/Weston kiosk 路径或对应 EGL 输出路径），不能假定标准 KMS connector 可用。

## 3. 显示链路职责

| 层级 | 职责 | 本项目选择 |
| --- | --- | --- |
| systemd | 开机启动、异常重启、日志 | 一个 HMI 主服务 |
| Qt Quick | 现代 UI、触摸交互、页面与状态管理 | Qt 6，业务逻辑使用 C++ |
| Qt Scene Graph / GPU | UI 合成、动态画面纹理、动画 | 优先使用 Qt 6 默认 RHI 后端 |
| EGLFS | 无 X11/Wayland 桌面的全屏输出 | EGLFS 的 DRM/KMS 集成后端 |
| DRM/KMS | 显示器检测、分辨率、buffer 提交、垂直同步 | Linux 内核显示接口 |
| HDMI | 物理信号输出 | 由 DRM/KMS 驱动控制 |
| libinput / Qt 输入系统 | 后续触摸输入 | 触摸屏接入后验证 |

Qt 的 EGLFS 不是普通桌面窗口：它让单个全屏应用直接使用 Linux 图形设备输出画面。设备仍然没有桌面、任务栏或窗口管理器。

## 4. DMA、DMA-BUF 与性能

### 4.1 DMA 的正确定位

DMA（Direct Memory Access）允许硬件在内存与设备之间传输数据，而无需 CPU 对每个字节参与搬运。显示控制器扫描 framebuffer 并输出 HDMI，本身也依赖 DMA 类硬件传输。

DMA-BUF 是 Linux 中跨驱动共享图像缓冲区的机制。典型用途是把相机、视频解码器、GPU 或其他硬件生成的 buffer 交给显示/合成模块，而不先复制成普通 CPU 内存图像。

### 4.2 哪些情况 DMA-BUF 最有价值

| 动态画面来源 | 优先路径 | 预期收益 |
| --- | --- | --- |
| 相机或硬件视频解码器 | V4L2/GStreamer → DMA-BUF → GPU/KMS | 避免 CPU 全帧复制和颜色转换 |
| GPU 生成的画面 | GPU render target → KMS buffer | 避免读回 CPU，再上传 GPU |
| 外部硬件模块的图像 buffer | DMA-BUF 导入为纹理或 KMS Plane | 低延迟、低 CPU 占用 |
| C++ 在 CPU 内存中逐像素绘制 | 预分配双缓冲 → GPU 纹理上传 | DMA-BUF 通常不能消除 CPU 生成像素的成本 |

若画面完全由 CPU 算法生成，DMA 不能让算法计算更快。需要优先评估是否能将画面生成、颜色变换或几何绘制迁移到 GPU shader；否则采用预分配缓冲区与受控帧率，避免额外分配和复制。

### 4.3 性能原则

- 避免每帧创建新的 `QImage`、`QPixmap` 或大块临时内存。
- 避免 CPU 图像 → GPU 纹理 → CPU 图像的往返路径。
- 以 page flip/vblank 驱动呈现节奏，避免无上限渲染循环。
- 页面静止时不主动持续刷新；动态区域单独按实际帧率更新。
- 先测量再优化：记录端到端延迟、帧时间、掉帧、CPU 占用和内存带宽。

1920×1080 RGBA8888 单帧约为 7.9 MiB；若 CPU 每秒生成并上传 60 帧完整图像，仅单向数据量约为 475 MiB/s。因此，1080p 全帧 60 FPS 的 CPU 图像路径必须谨慎验证。

## 5. 推荐软件架构

```text
┌─────────────────────────────────────────────────────┐
│ systemd：启动、重启策略、日志                         │
└─────────────────────┬───────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────┐
│ 单一 HMI 主进程（独占显示设备）                       │
│  ├─ C++ 业务层：设备通信、状态机、数据处理            │
│  ├─ Qt Quick：界面、按钮、告警、页面导航              │
│  ├─ DynamicFrameItem：动态画面显示区域                │
│  ├─ Frame Producer：CPU、GPU 或外部硬件图像生产者     │
│  └─ Input：后续接入触摸事件                           │
└─────────────────────┬───────────────────────────────┘
                      │ EGLFS / DRM-KMS
┌─────────────────────▼───────────────────────────────┐
│ HDMI 显示器                                           │
└─────────────────────────────────────────────────────┘
```

其他未来业务进程不应直接打开 `/dev/dri/card*`。它们应通过 Unix domain socket、共享内存或其他 IPC 向 HMI 主进程提供状态或帧数据，避免多个进程争夺 DRM master。

## 6. 动态画面实现策略

### 6.1 第一版：CPU 双缓冲纹理

适用于先验证“程序每帧生成画面”的需求。

1. C++ 工作线程在预分配的双/三缓冲中生成下一帧。
2. 缓冲完成后以线程安全队列通知 UI。
3. Qt Quick 动态组件在渲染阶段更新纹理。
4. Scene Graph 将动态纹理与按钮、文字、状态栏合成并提交显示。

禁止在 UI 线程中执行整帧重计算，也不要在每帧分配新的图像对象。

### 6.2 性能版：GPU 直接绘制

若需要稳定的 1080p 60 FPS，或动态画面包含大量透明叠加、缩放和实时视觉效果，应把画面生成迁移到 Qt Quick Scene Graph 的 shader/自定义渲染路径。每帧只更新参数、少量顶点或必要纹理，而不是上传整张 CPU 图像。

新项目应优先保持 Qt 6 RHI 兼容；不要把 OpenGL 专用的 `QQuickFramebufferObject` 作为长期核心架构。

### 6.3 零拷贝版：DMA-BUF

当图像来自 V4L2、硬件解码器、GPU 或其他支持 DMA-BUF 的生产者时，按目标平台驱动能力评估：

```text
生产者 buffer → DMA-BUF fd → EGL/Qt 图形层导入为纹理
                              或 → DRM overlay plane 直接扫描输出
```

若必须把画面和 UI 混合，通常导入 GPU 纹理后由 Qt Quick 合成更灵活；若画面是完整全屏且硬件 Plane 支持所需格式/缩放，direct scanout 的开销最低。DMA-BUF 方案需要目标设备的 GPU、V4L2 和 DRM 驱动共同支持，必须在真实硬件上验证。

## 7. UI 设计原则

界面采用现代化深色控制台风格，而非传统拟物化工控面板：

- 使用近黑或深灰背景、大留白、清晰的信息层级。
- 使用有限的品牌主色；红、黄、绿仅表达告警和设备状态。
- 采用统一圆角、间距和线性图标；触摸目标至少 48–56 px。
- 动画保持克制，建议 150–250 ms；优先确保信息可读性和操作反馈。
- 动态画面作为页面主区域，UI 控件以低干扰方式覆盖或排布在侧栏/底栏。

## 8. 分阶段验证计划

### 阶段 A：显示硬件探测

- 枚举 DRM connector、HDMI 状态、EDID、可用 mode、plane 和像素格式。
- 在无桌面环境下显示 KMS 测试图。
- 验收：重启后可稳定点亮 HDMI，正确选取目标分辨率。

### 阶段 B：Qt EGLFS 最小 HMI

- 运行一个全屏 Qt Quick 页面，包含现代卡片、按钮和状态组件。
- 由 systemd 开机启动并在异常退出时自动重启。
- 验收：无 X11/传统桌面，应用稳定独占屏幕。

### 阶段 C：动态帧基线

- 实现 CPU 双/三缓冲的动态帧组件。
- 在目标分辨率测试 30 FPS 和 60 FPS。
- 验收：记录帧时间、CPU、内存和端到端延迟；无明显撕裂或持续掉帧。

### 阶段 D：GPU / DMA-BUF 优化

- 对不达标的动态画面迁移到 GPU 渲染路径。
- 若生产者支持 DMA-BUF，验证纹理导入或 KMS Plane 路径。
- 验收：与 CPU 基线路径对比 CPU 占用、帧率和延迟，并保留可回退实现。

### 阶段 E：触摸与可靠性

- 接入触摸屏，验证点击、滑动、长按、多点触控（如设备支持）。
- 验证 HDMI 热插拔、显示器断电恢复和服务重启。
- 验收：连续运行、异常恢复和触摸交互满足实际设备要求。

## 9. 当前决策

1. 以 C++20 + Qt 6 Quick + EGLFS/DRM-KMS 构建无桌面 HMI。
2. 第一版动态画面采用预分配 CPU 双/三缓冲，先取得真实性能数据。
3. 将 DMA-BUF 作为硬件图像源的零拷贝优化路径，而不是所有画面的一般实现。
4. HMI 主进程独占显示设备，未来业务模块通过 IPC 接入。
5. 触摸屏到位后接入 Qt/Linux 输入系统，并保持显示与输入模块解耦。
